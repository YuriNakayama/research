# 自動リサーチパイプライン — リスクと依存関係

## リスク一覧

| # | リスク | 影響 | 発生確率 | 軽減策 |
|---|--------|------|----------|--------|
| 1 | Claude Maxセッションの期限切れ | High | Medium | EFS上の認証情報の有効期限を監視。期限切れ時はSlack通知でアラート。定期的な手動再ログインの運用手順を整備 |
| 2 | Claude CLIの出力が不安定（ハルシネーション、フォーマット崩れ） | Medium | Medium | 出力ファイルの基本的なバリデーション（空ファイルチェック、最小行数チェック）を実装。PRレビューで人間が確認 |
| 3 | NATゲートウェイのコスト超過 | Medium | Low | シングルAZ構成で月額約$30-50に抑制。CloudWatchでデータ転送量を監視 |
| 4 | GitHub Appトークンの有効期限（1時間） | Medium | Low | リサーチ実行時間を30分以内に制限。超過時はトークン再取得ロジックを追加 |
| 5 | Claude CLI のバージョンアップによる破壊的変更 | Medium | Low | Dockerfileでバージョン固定（`npm install -g @anthropic-ai/claude-code@x.y.z`）。定期的な更新確認 |
| 6 | EFSのファイル破損・消失 | High | Low | EFSのバックアップポリシー設定。認証情報の再作成手順を整備 |
| 7 | Fargateタスクのタイムアウト | Low | Low | EventBridgeのリトライポリシー（最大2回）で対応。CloudWatch Logsで実行時間を監視 |
| 8 | Slack Webhook URLの無効化 | Low | Low | タスク本体（リサーチ+PR）とSlack通知は独立。通知失敗でもリサーチ結果は保存される設計 |

## 外部依存

| 依存先 | 種類 | リスク |
|--------|------|--------|
| Claude Max（Anthropic） | サービス | セッションベース認証のため、認証方式の変更に影響を受ける |
| GitHub API | サービス | API レート制限（5000 req/h）。本ユースケースでは問題なし |
| Slack Incoming Webhook | サービス | Slack側の仕様変更。代替通知手段の検討が容易 |
| npm @anthropic-ai/claude-code | パッケージ | メジャーバージョンアップ時の互換性 |
| gh CLI | パッケージ | GitHub公式、安定性は高い |
| AWS ECS Fargate | インフラ | AWSの安定したマネージドサービス。リスク低 |

## テクニカルデット

| 項目 | 説明 | 将来の対応 |
|------|------|-----------|
| 単一プロンプト実行 | 初期は1プロンプト/1タスク | 設定ファイルで複数プロンプトの順次/並列実行に対応 |
| Claude Maxセッション管理 | 手動ログイン前提 | セッション期限の自動検知と通知の仕組み |
| スキル非対応 | 初期はスキルなしで実行 | config.yamlの`skill`フィールドで切り替え可能な設計は済み |
| テスト環境 | ローカルモックテストのみ | ステージング環境でのE2Eテスト |
| NATゲートウェイ（シングルAZ） | 可用性よりコスト優先 | 必要に応じてマルチAZ化 |
