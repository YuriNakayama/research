# NL2SQL-BUGs: A Benchmark for Detecting Semantic Errors in NL2SQL Translation

- **Link**: https://arxiv.org/abs/2503.11984
- **Authors**: Xinyu Liu, Shuyu Shen, Boyan Li, Nan Tang, Yuyu Luo
- **Year**: 2025
- **Venue**: SIGKDD 2025
- **Type**: Academic Paper

## Abstract

Natural Language to SQL (NL2SQL) systems have the potential to democratize data access by enabling users to interact with databases using natural language rather than writing SQL. While existing approaches primarily focus on improving the generation accuracy of NL2SQL systems, few efforts have been directed toward identifying semantic errors in the generated SQL queries. To address this gap, we introduce NL2SQL-BUGs, the first comprehensive benchmark for evaluating the detection of semantic errors in NL2SQL translation. We propose a two-level taxonomy to systematically classify semantic errors, covering 9 main categories and 31 subcategories. Using this taxonomy, we create a dataset with 2,018 expert-annotated instances. Through extensive experiments, we find that the average detection accuracy of current large language models is only 75.16%. Furthermore, our method has successfully identified 106 previously undetected annotation errors within BIRD, a widely-used NL2SQL benchmark, demonstrating its practical value for improving benchmark quality.

## Abstract（日本語訳）

自然言語からSQL（NL2SQL）への変換システムは、ユーザーがSQLを書く代わりに自然言語でデータベースと対話できるようにすることで、データアクセスの民主化を実現する可能性を持つ。既存のアプローチは主にNL2SQLシステムの生成精度の向上に焦点を当てているが、生成されたSQLクエリにおけるセマンティックエラーの特定に向けた取り組みはほとんどない。このギャップに対処するため、我々はNL2SQL-BUGsを導入する。これはNL2SQL変換におけるセマンティックエラーの検出を評価するための初の包括的ベンチマークである。我々は2レベルの分類体系を提案し、セマンティックエラーを9つの主要カテゴリと31のサブカテゴリに体系的に分類する。この分類体系を用いて、2,018件の専門家アノテーション付きインスタンスからなるデータセットを作成した。広範な実験を通じて、現在の大規模言語モデルの平均検出精度はわずか75.16%であることが判明した。さらに、我々の手法はBIRDという広く使用されているNL2SQLベンチマークにおいて、これまで検出されていなかった106件のアノテーションエラーを特定することに成功し、ベンチマーク品質向上における実用的価値を実証した。

## 概要

NL2SQL-BUGsは、NL2SQL変換において生成されるSQLクエリのセマンティックエラー（意味的誤り）を体系的に検出・分類するための初の専門的ベンチマークである。従来のNL2SQL研究は主に生成精度の向上に注力してきたが、生成されたSQLが構文的には正しいものの意味的に誤っているケースへの対処は見過ごされてきた。セマンティックエラーは構文エラーと異なり実行時にエラーを起こさないため検出が困難であり、誤った分析結果やビジネス上の意思決定ミスにつながるリスクがある。本研究は、9つの主要カテゴリと31のサブカテゴリからなる2レベルの分類体系を提案し、2,018件の専門家アノテーション付きインスタンスを構築した。正解例1,019件と誤り例999件を含むバランスの取れたデータセットにより、現在のLLMの検出能力を包括的に評価している。実験の結果、GPT-4oとClaude-3.5-Sonnetが最もバランスの取れた性能を示したものの、全体の平均検出精度は75.16%にとどまり、特にサブクエリ関連エラーやデータベース知識を要するエラーの検出が困難であることが明らかになった。さらに、BIRDデータセットにおける106件（6.91%）およびSpiderにおける16件（1.55%）の未検出アノテーションエラーの発見は、既存ベンチマークの品質改善にも貢献する重要な成果である。

## 問題設定

- **セマンティックエラーの見過ごし**: 既存のNL2SQL研究は生成精度（Execution Accuracy, Exact Match）の向上に焦点を当てており、生成されたSQLが構文的に正しく実行可能であっても意味的に誤っているケースの検出は十分に研究されていない。

- **検出困難性**: セマンティックエラーは構文エラーと異なりデータベースエンジンによって自動検出されないため、ユーザーが誤った結果を正しいと誤認するリスクが高い。これは実運用環境において深刻な問題となる。

- **分類体系の不在**: セマンティックエラーの種類を網羅的に整理した分類体系が存在せず、エラーの性質に応じた検出戦略の開発が困難であった。

- **ベンチマークデータの品質問題**: 広く使用されているBIRDやSpiderなどのNL2SQLベンチマーク自体にもアノテーションエラーが含まれている可能性があり、評価の信頼性に影響を及ぼしている。

- **LLMの検出能力の未検証**: 大規模言語モデルがセマンティックエラーをどの程度検出できるかについての体系的な評価が行われていない。

## 提案手法

### ベンチマーク設計

**データセット構成**:
- 総インスタンス数: 2,018件
- 正解例: 1,019件（正しいNL-SQLペア）
- 誤り例: 999件（セマンティックエラーを含むSQLクエリ）
- 各インスタンスは自然言語クエリ（NL）、データベーススキーマ（DB）、生成SQLクエリ、正確性ラベル、詳細なエラーアノテーションで構成

**2レベルセマンティックエラー分類体系（9主要カテゴリ・31サブカテゴリ）**:

1. **属性関連エラー（Attribute-related）**: データベースカラムへの属性マッピングの問題
   - 属性ミスマッチ（Attribute Mismatch）
   - 属性冗長（Attribute Redundancy）
   - 属性欠落（Attribute Missing）

2. **テーブル関連エラー（Table-related）**: データベーステーブルの不正な処理
   - テーブルミスマッチ（Table Mismatch）
   - テーブル冗長（Table Redundancy）
   - テーブル欠落（Table Missing）
   - JOIN条件ミスマッチ（Join Condition Mismatch）
   - JOINタイプミスマッチ（Join Type Mismatch）

3. **値関連エラー（Value-related）**: クエリ値の誤解釈・フォーマットエラー
   - 値ミスマッチ、値フォーマット、値欠落など

4. **演算子関連エラー（Operator-related）**: 比較演算子・論理演算子の誤適用
   - 比較演算子ミスマッチ、論理演算子エラーなど

5. **条件関連エラー（Condition-related）**: 明示的・暗黙的クエリ条件の問題
   - 条件欠落、条件冗長、条件ミスマッチなど

6. **関数関連エラー（Function-related）**: SQL関数（集約、ウィンドウ、日時）の誤用
   - 集約関数エラー、日時関数エラーなど

7. **句関連エラー（Clause-related）**: GROUP BY、HAVINGなどのSQL句の誤管理
   - GROUP BY欠落、ORDER BYエラーなど

8. **サブクエリ関連エラー（Subquery-related）**: サブクエリの欠落・不一致
   - サブクエリ欠落、サブクエリ構造ミスマッチなど

9. **その他のエラー（Other）**: 上記に分類されないエラー

### 評価指標

- **検出精度（Detection Accuracy）**: セマンティックエラーの有無を正しく判定できた割合
- **正例再現率（Positive Recall）**: 実際にエラーを含むSQLを正しくエラーと判定できた割合
- **負例再現率（Negative Recall）**: 正しいSQLを正しく正解と判定できた割合
- **エラータイプ別検出率**: 各エラーカテゴリに対する検出精度

## 実験・評価

### セットアップ

- 評価対象モデル: GPT-4o、Claude-3.5-Sonnet、Gemini-2.0-Flashを含む複数のLLM
- 評価タスク: 与えられた自然言語クエリ、データベーススキーマ、生成SQLに対してセマンティックエラーの有無を判定
- データセット: 2,018件の専門家アノテーション付きインスタンス（正解1,019件、誤り999件）
- アノテーション: 複数の専門家による詳細なエラーラベリングとカテゴリ分類

### 主要結果

**全体的な検出精度**:
現在のLLMの平均セマンティックエラー検出精度は75.16%であり、実用的な自動検出システムとしては不十分なレベルにある。

**モデル別性能**:
- GPT-4oが最も高い検出精度（約78%）を達成し、全エラータイプにわたって安定した性能を示した
- Claude-3.5-Sonnetも同様にバランスの取れた性能を発揮
- Gemini-2.0-Flashは正例再現率が高いものの、正しいSQLを誤ってエラーと判定する偽陽性が多い

**エラータイプ別分析**:
- 条件関連エラーと値関連エラーの検出精度が最も高い
- サブクエリ関連エラーとデータベース深層知識を要するエラーの検出が最も困難
- 関数関連エラー（特に集約関数の誤用）の検出も課題

**既存ベンチマークの品質検証**:
- BIRDデータセットにおいて106件（6.91%）の未検出アノテーションエラーを発見
- Spiderの開発セットにおいて16件（1.55%）のセマンティックエラーを特定
- これらの発見は既存ベンチマークの信頼性に疑問を投げかけ、品質改善の必要性を示す

## 備考

- NL2SQL-BUGsはセマンティックエラー検出という新しい研究方向を切り開く先駆的なベンチマークであり、NL2SQLシステムの実用化における品質保証の重要性を浮き彫りにしている
- 9カテゴリ・31サブカテゴリの分類体系は、今後のエラー検出手法の開発において統一的な評価基準となりうる
- BIRDおよびSpiderにおけるアノテーションエラーの発見は、ベンチマーク駆動型研究において「ベンチマーク自体の品質」という見落とされがちな問題を提起している
- 平均検出精度75.16%という結果は、セマンティックエラー検出が生成タスクとは質的に異なる能力を要することを示唆しており、専門的な検出モデルの開発が必要である
- SIGKDD 2025に採択されており、データベース・データマイニングコミュニティにおける高い評価を受けている
