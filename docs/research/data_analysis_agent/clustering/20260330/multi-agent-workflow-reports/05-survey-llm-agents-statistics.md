## Abstract (English)

In recent years, data science agents powered by Large Language Models (LLMs), known as "data agents," have shown significant potential to transform the traditional data analysis paradigm. This survey provides an overview of the evolution, capabilities, and applications of LLM-based data agents, highlighting their role in simplifying complex data tasks and lowering the entry barrier for users without related expertise. We explore current trends in the design of LLM-based frameworks, detailing essential features such as planning, reasoning, reflection, multi-agent collaboration, user interface, knowledge integration, and system design, which enable agents to address data-centric problems with minimal human intervention. Furthermore, we analyze several case studies to demonstrate the practical applications of various data agents in real-world scenarios. Finally, we identify key challenges and propose future research directions to advance the development of data agents into intelligent statistical analysis software.

## 概要

本論文は、統計学とデータサイエンスの観点からLLMベースのエージェントを体系的に調査した初のサーベイであり、統計学分野の権威ある学術誌The American Statisticianに掲載されている点が特筆される。データ分析の4つの障壁を特定し、それらを克服するための7つのフレームワーク設計要素を体系化している。さらに、Wine QualityデータセットとAuto MPGデータセットを用いた実践的なケーススタディにより、データエージェントの現在の能力と限界を具体的に示している。

## 問題と動機

### データ分析における4つの障壁

本論文は、データ分析の民主化を妨げる4つの根本的な障壁を特定する。

## 提案手法

### 7つのフレームワーク設計要素

本論文は、LLMベースのデータエージェントフレームワークに不可欠な7つの設計要素を体系化する。

#### 要素1: プランニング（Planning）

2つの主要アーキテクチャが存在する。

```
線形構造プランニング:
  タスク → ステップ1 → ステップ2 → ステップ3 → 結果

階層構造プランニング:
  タスク
  ├── サブタスク1
  │   ├── サブサブタスク1a
  │   └── サブサブタスク1b
  ├── サブタスク2（動的追加）
  └── サブタスク3
      └── 依存関係に基づく適応的実行
```

| プランニングタイプ | 特徴 | 適用場面 |
|------------------|------|---------|
| **線形構造** | 逐次的分解、シンプル | 明確な手順のタスク |
| **階層構造** | 動的・適応的、複雑な依存関係に対応 | 複雑なデータ分析ワークフロー |

具体的な推論手法として、Chain-of-Thought、ReAct、Tree-of-Thoughts、Graph-of-Thoughtsが挙げられる。

#### 要素2: 推論（Reasoning）

LLMが論理的推論と段階的な思考を通じて、データ分析の各段階における意思決定を行う。統計的手法の選択、パラメータの決定、結果の解釈において、推論能力が中核的な役割を果たす。

#### 要素3: リフレクション（Reflection）

```
実行 → 結果評価 → エラー検出 → 自己修正 → 再実行
  ↑                                         │
  └─────────── 反復的改善ループ ──────────────┘
```

エージェントが過去のアクションを評価し、自己修正メカニズムを通じて改善する。軌跡フィルタリングによりエージェントはエラーから学習し、反復的な修正サイクルを通じてデバッグ能力を向上させる。

#### 要素4: マルチエージェント協調（Multi-Agent Collaboration）

役割割り当てによるタスク分解で、エージェントがコミュニケーションし集合的パフォーマンスを最適化する。

| エージェント役割 | 機能 |
|----------------|------|
| **プログラマーエージェント** | コード生成・実行 |
| **インスペクターエージェント** | コードレビュー・品質検証 |
| **データエージェント** | データ前処理・統計分析 |

#### 要素5: ユーザーインターフェース（User Interface）

4つのカテゴリが特定されている。

| UIタイプ | 説明 | 代表例 |
|---------|------|--------|
| **IDEベース** | Jupyter環境との統合 | Jupyter-AI, Chapyter |
| **独立システム** | ChatGPTスタイルのインターフェース | ChatGPT-ADA, LAMBDA |
| **コマンドラインベース** | CLI経由の操作 | 上級ユーザー向け |
| **OSベース** | システムレベルのインタラクション | Spider2-V |

#### 要素6: 知識統合（Knowledge Integration）

3つのアプローチが存在する。

1. **ツールとしてのコード**: 専門家コードを呼び出し可能なツールとして扱う
2. **RAG（Retrieval-Augmented Generation）**: 関連コードをコンテキストに埋め込む
3. **外部知識アクセス**: 外部知識ソースへのアクセス

LAMBDAフレームワークは、分析コードを記述と実行可能コンポーネントに解析し、知識ベースに格納する先進的なアプローチを採用。

#### 要素7: システム設計（System Design）

```
┌──────────────────────────────────────────────┐
│           データエージェントシステム            │
│                                               │
│  ┌──────────┐     ┌──────────────────┐       │
│  │ LLMコア   │     │ サンドボックス    │       │
│  │           │     │ 実行環境         │       │
│  │ ・Planning│────→│                  │       │
│  │ ・Reason- │     │ ・Python         │       │
│  │   ing     │←────│ ・R              │       │
│  │ ・Reflect-│     │ ・SQL            │       │
│  │   ion     │     │ ・Jupyter        │       │
│  │ ・Error   │     │                  │       │
│  │  Handling │     └──────────────────┘       │
│  └──────────┘                                │
│       ↕                                       │
│  ┌──────────────────────────────────────┐    │
│  │ ワークフロー自動化・可視化生成         │    │
│  └──────────────────────────────────────┘    │
└──────────────────────────────────────────────┘
```

フレームワークアーキテクチャはワークフロー自動化、可視化生成、および実行環境（サンドボックス）との統合を重視する。

## 実験と評価

### ケーススタディ1: Wine Qualityデータセット（EDA）

**データセット**: 4898サンプル x 11特徴量

ChatGPTとLAMBDAによるWine Qualityデータの探索的データ分析（EDA）を実施。アルコール含有量がワイン品質（赤ワイン・白ワイン）に与える影響を分析し、モデル構築とレポート生成を含む包括的な分析を行った。

**評価結果:**
- 両エージェントとも基本的なEDAタスク（記述統計、相関分析、可視化）を適切に実行
- データの前処理と特徴量選択において、エージェントごとに異なるアプローチを採用
- 可視化の品質と解釈可能性に差異が見られた

### ケーススタディ2: Auto MPGデータセット（線形回帰）

**データセット**: 398サンプル x 7特徴量

馬力と車重を説明変数として燃費（miles-per-gallon）を予測する線形回帰モデルの構築。残差診断およびBreusch-Pagan異分散性検定を含む厳密な統計的推論を実施。

**評価結果:**
- モデル適合、残差プロット生成、統計的検定の実行を正確に遂行
- エージェントが厳密な統計的推論（異分散性検定）を実行できることを実証
- 結果の解釈において、統計学的に正確な記述が可能

### ケーススタディ3: Wine Qualityデータセット（ブートストラップ信頼区間）

赤ワインの平均アルコール含有量に対するブートストラップ信頼区間推定を実施。1000回のリサンプリングによる95%信頼区間の算出。

**評価結果:**
- ブートストラップ法の正しい実装と信頼区間の適切な計算
- 統計的に有効な推定手法の自動実装を実証
- 結果の解釈と報告の品質が高い

## 備考

- The American Statisticianという統計学の権威ある学術誌に掲載されている点は、LLMベースのデータエージェントが統計学コミュニティからも注目されていることを示す
- 4つの障壁の特定は、データ分析の民主化という文脈において、技術的課題だけでなく教育的・組織的課題も含む包括的な問題定義を提供
- 7つのフレームワーク設計要素は、実装者がデータエージェントを構築する際のチェックリストとして機能する
- ケーススタディにおけるBreusch-Pagan検定やブートストラップ信頼区間推定の実行は、エージェントが高度な統計的手法を適用できることの具体的な証拠
- 23のデータエージェントの比較表（Table 1）は、フレームワーク選択の際の実用的なガイドとして価値が高い
- 論文の焦点が統計学とデータサイエンスに特化しているため、ソフトウェア開発やゲームシミュレーション等の応用は対象外であり、他のサーベイと相補的な関係にある
- 商用システム（ChatGPT-ADA、GPT-4o）と研究フレームワーク（LAMBDA、Data Interpreter）の両方をカバーしている点が実務的に有用
