## 0. 勝つための3つのレバー（重要度順）

| # | レバー | なぜ重要 | 具体策 |
|---|---|---|---|
| **①** | **デッキ選択** | 同じ操縦コードでも順位が大きく動く。波は1日で変わる | 現環境上位（Trevenant/Alakazamを叩ける）デッキをクローン。**操縦しやすいシンプルな構築**を優先 |

| **③** | **実ラダーでのA/B検証** | **ローカルsim（cabt）はラダー順位を当てない** | 1日5提出・最新2件採点を使い、**毎日のメタ再分析＋divergence分析**で改善 |

## 1. 最初の意思決定：ML か ルールベースか

**結論：ルールベース一択（少なくとも序盤〜中盤戦）**

- 参考実装・他の参加者の経験則: **PPO/模倣学習に走った試みの多くが失敗**。理由 — 隠れ情報＋ランダム性＋巨大な行動空間で、限られた計算資源のRLは安定しない。模倣学習パイプラインは参考実装でも「廃案」とされた。
- **ルールベースの強み**: 解釈可能（Strategyレポートで説明しやすい）、デバッグ可能、クラッシュさせない設計が容易、divergence分析で人間トップに漸近できる。
- **MLを使うなら**: ルールベースが頭打ちになってから、**スコアの再ランク**や**特定コンテキストの方策**に限定して載せる程度。土台はルールベース。

> Strategyトラックは「安定性・デッキ設計思想・Simulation成績」で評価される。**説明できないブラックボックスより、設計意図が明快なルールベースの方がレポートで戦える**。

## 2. 実装ロードマップ（フェーズ）

### Phase 0 — 環境セットアップ（0.5日）
- Kaggle公式から starter（エンジン `cg-lib`・サンプルNotebook・カードデータ・ルール）を入手。
- `kaggle-environments==1.30.1`（ラダーと同バージョン）をpip。
- `agent(obs_dict)` の**最小提出**を作って1回通す（公式サンプルのDragapultをそのまま）。**まず「動いて落ちない」状態を作る**。
- 参考実装 [`github.com/wmh/ptcg-abc`](https://github.com/wmh/ptcg-abc) を読み、`policy_base.py` のスカフォールド（2段クラッシュガード・PrizeTracker・normalize_selection）を土台に借りる。

### Phase 1 — 「落ちないサンプル」をラダーに乗せ、基準を作る（1日）
- 公式Dragapultサンプル＋堅牢ラッパーを提出 → 初期Eloを把握。
- `tools/check_agent.py` 相当で**過充填0・クラッシュ0**を保証する自動チェックを用意（これが Strategy評価の「安定性」に直結）。

### Phase 2 — メタを読み、デッキを選ぶ（1〜2日、以後毎日）
- 日次 episodes をDLし `meta_analyze.py --elo 1150` で**アーキタイプ分布・勝率・相性行列**を出す。
- **デッキ選択基準**（下記チェックリスト）でデッキを1〜2個に絞る。現状の有力解は **Dragapult ex**（公式サンプルあり＝初期品質高）と **Mega Starmie ex**（#1構築・両巨頭を叩く）。
- ヘッジに **Alakazam**（全1プライズ）を1枚持つ。

### Phase 3 — divergence駆動でカード毎スコアを校正（継続）
- トップ人間（Elo≥1150、できれば#1プレイヤー名指定）の棋譜を自分のagentで再生 → `divergence_decode` で**コンテキスト別に「人間の手 vs 自分の手」を集計**。
- 食い違いの多いカード/技から直す。**鉄則: スコアを闇雲に上げない。過剰優先しているカードを下げる**（実例: Fezandipiti 53000→35000で改善。逆にEVOLVEを50000に上げたら悪化）。
- 各変更は **`check_agent` PASS → cabt≥80局で回帰なし → 実ラダーで採点**の順で確定。

### Phase 4 — メタ追従と最終化（締切前）
- Simulation締切（〜8/17）まで、毎日メタを見てデッキ/操縦を微調整。
- 並行して **Strategyレポート**を書き上げる（〜9/14）。Simulationの最終順位・相性データ・設計思想を盛り込む。

## 3. デッキ選択チェックリスト（6項目）

1. **弱点 vs 環境**: 主力アタッカーの弱点が、現環境の主要タイプに刺さらないか。
2. **非ex対策**: Crustle等のex免疫（ID `{158,207,330,345}`）を**非exアタッカーで割れるか**。
3. **タイプ被覆**: 環境主力の弱点を突けるか（ほのお→Crustle、エスパー→Lucario 等）。
4. **プライズ経済**: 全1プライズ（Alakazam/Trevenant）はex/Mega ex相手に有利。2〜3プライズ（Dragapult/Mega Starmie）は**テンポ/コントロールで上回る**設計か。
5. **ドロー/サーチ/加速の安定性**: 事故率が低いか（**ボットは複雑な事故処理が苦手**）。
6. **ロック対策**: 「相手の次ターンをロックする大技」を持つなら、ロックされた時の**入替/別アタッカー**があるか。

> **+0番目の隠れ基準: シンプルさ**。ボットが正確に回せるか。単純1エンジン＞複雑2進化リサイクル。

## 4. 操縦（政策）の鉄則 — 参考実装が血を流して学んだ教訓

0. **【最重要・公式サンプルで3-0確定／cluster-08】スコアは「強さ」でなく「ターン内の処理順序」で付ける**。進化を高スコア・攻撃を最終手として低スコア（ATTACKは`score=attackId`の低値）にする。理由は「攻撃が弱いから」ではなく「攻撃はターンを終える＝最後に行うべき行動だから」。進化→特性→エネ付け→攻撃 の順に高→低。**これがルールベースを"綺麗に操縦"する核心原則**。
1. **エネルギー規律**: 各技の**実コストから導出**して「どの技も撃てない間だけ」エネを付ける。過充填は構造的にゼロにする。
2. **真の勝ち筋技を見極める**: 派手な大技でなく**安いワークホース技**が勝ち筋なことが多い（Mega Starmieは Jetting Blow[水1] が主力、Nebula[CCC]ではない）。
3. **使い切りエネ（Ignition）の扱い**: ターン終了時トラッシュ系は**そのターン攻撃する前のポケにだけ**。ビルドアップ/ベンチ付け厳禁。
4. **DISCARDスコアラーは勝ち筋ラインを死守**: Ultra Ballのコストで主力/進化起動カードを捨てない。捨てるのは最も潤沢な余剰（基本エネ）。
5. **スプレッド/スナイプは低HPの開発パーツを狙う**: ドローエンジン・進化基礎（Dunsparce/Phantump/Abra）を潰す。**KOできない高HP壁に撃ち込まない**（150HPスノーラックスに78ダメージを無駄撃ちしたバグ）。
6. **ドローエンジンデッキは積極的に引く**: 「デッキ切れガード」を足すと必ず悪化（60%→38%）。大きな手札が勝ち筋（Alakazam）。
7. **先攻/後攻はデッキ依存**: 必ず棋譜から読む。仮定しない。
8. **基礎切れ緊急時の選牌**: 手札・ベンチにたねが無いのに進化サーチ（Hilda）を打たない。たねを取れるカード（Buddy Poffin/Ultra Ball）を優先。
9. **プライズトラッカー**: 見えているカードからサイド落ちを推定し、サイド落ちした勝ち筋を掘りに行かず、**残っている勝ち筋で計画**する。
10. **取り過ぎない**: プライズの取り過ぎは相手に反撃ドロー（Unfair Stamp）を与える。

## 5. 検証の規律（最重要の落とし穴）

```
変更 → check_agent（過充填0/クラッシュ0）→ cabt ≥80局（回帰なし）→ 実ラダー採点
                                                  ▲                        ▲
                                       40局は±10ptノイズ。            ★唯一の真の審判★
                                       回帰検出にのみ使う          （ローカルは順位を誤予測）
```

- **cabt ≠ ラダー**: ローカルが順位を逆転させた実例多数（Bellibolt 836 > Quilava 532 だがcabtは逆）。**cabtは回帰検出（バグ取り）の道具であって戦略のオラクルではない**。
- **提出枠（1日5・最新2採点）を浪費しない**: divergence検証を通したものだけ出す。
- **アノマリー駆動**で直す（勝率の数字を追わない）: `attack_no_damage`（0ダメージ技を撃った）/ `no_offense_loss`（一度も圧力をかけられず負け）/ `deckout_loss` / `error_games`（クラッシュ）。これらは**決定論的で確実に直る**。

## 6. Strategyトラックで勝つためのレポート構成案

評価軸（安定性・デッキ設計思想・Simulation成績）に沿って:

1. **デッキ設計の思想**: なぜこのデッキか（メタ分析・相性表・プライズ経済の論証）。
2. **操縦アルゴリズム**: `agent(obs_dict)` の構造、コンテキスト別スコアリング、エネ規律の「過充填不可能性」、クラッシュ0設計。
3. **データ駆動の校正過程**: divergence分析で人間トップに漸近させた具体例（before/after数値）。
4. **安定性の証拠**: `check_agent` のPASS実績、フォールバック率0、長期ラダーでのEloの安定。
5. **Simulationの裏付け**: 最終順位・主要相性の勝率。
6. **メタ追従の運用**: 日次パイプライン（autopsy）で環境変化に追従した記録。

> **差別化ポイント**: 「ブラックボックスではなく、**全意思決定が説明可能**な設計」「**実データ（棋譜）で校正**した」「**安定性を構造的に保証**した」の3点は、Strategy評価で強い物語になる。

## 7. 90秒サマリー（経営層向け）

- **何を作るか**: ポケカ標準戦を戦う `agent(obs_dict)` 1関数。エンジン（cabt）は支給。**思考プレイヤーを書く問題**。
- **どう勝つか**: ①現環境を叩ける**シンプルで操縦しやすいデッキ**を選び、②**カード毎に明示スコアを書いたルールベース**で正確に回し、③**実ラダーA/B＋トップ棋譜のdivergence分析**で毎日改善。
- **やらないこと**: いきなりRL/模倣学習。ローカルsimの勝率を信じ込む。提出枠を当てずっぽうで浪費。
- **狙い**: Strategyトラック上位8（各$30,000＋東京決勝）。そのための前提条件がSimulationの好順位。
- **最大のリスク**: メタが日替わりで激変すること → **日次のメタ再分析を運用化**して吸収する。
