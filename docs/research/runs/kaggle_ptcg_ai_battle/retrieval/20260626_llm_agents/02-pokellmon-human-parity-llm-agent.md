# PokeLLMon: 学習不要のin-context強化と知識拡張で人間並みに戦うLLMエージェント

## 概要

PokeLLMon(Sihao Hu, Tiansheng Huang, Ling Liu, arXiv:2402.01118)は、戦術バトルゲームで初めて人間並み(human-parity)の性能を達成したLLMエージェント。Pokemon Showdownのオンライン対戦で実プレイヤー相手にラダー49%・招待プレイヤー相手56%勝率を記録した。**モデルを一切再学習せず、プロンプト工学とin-context学習のみ**で達成している点が本コンペにとって重要で、限られた開発期間でLLMエージェントを構築する際の実装パターン集として価値が高い。

## 手法の要点

PokeLLMonは3つの戦略で構成される。

1. **In-Context強化学習(ICRL)**: モデル学習なしで、直前ターンの結果をテキストフィードバックとして即座に与え行動生成を逐次改善する。フィードバックは(a)連続ターンのHP変化、(b)技の有効性(効果抜群/今ひとつ/無効)、(c)技の優先度(先制の成否)、(d)実際の効果(能力変化・状態異常)の4種。
2. **知識拡張生成(KAG)**: 幻覚(hallucination)対策に外部知識を検索注入する。タイプ相性表(例: 草に強く炎に弱い)と、ポケデックスDBから取得した技・特性の詳細記述を与え、序盤の致命的ミスを削減する。
3. **一貫した行動生成(Consistent Action Generation)**: 強敵に対しCoT推論が毎ターン別ポケモンに交代し続ける「パニック交代(panic switching)」を、自己一貫性投票(k=3)で多数決を取り抑制する。

盤面は「自チーム属性(場/控え)・相手チーム情報(一部非公開)・場の状態(天候・地形・設置)・履歴ログ」の4要素に構造化テキスト変換される。**人間のようなjust-in-time意思決定**を実現している。

## 主要な結果や知見

- ラダー49%(105戦)、招待プレイヤー56%(50戦)、ヒューリスティックbotに58%(ベースラインの26%から大幅改善)。
- KAGとICRLの併用で序盤のタイプ相性ミスが大幅減少。タイプ相性の明示注入が特に効果大。
- **明確な弱点**: 人間の消耗(attrition)戦術に対し勝率18.75%。複数ターンにまたがる長期計画が必要な場面で、目先の盤面分析に偏るため弱い。
- 相手の行動予測を欺く読み合いにも脆弱で、「現在状態の分析」中心で「相手の次手予測」が弱いと自己分析している。

## 本コンペ(PTCG AI Battle)への応用

- **obs_dict→構造化テキスト変換器の設計図**: PokeLLMonの4要素分解は、`obs_dict`の`logs`(履歴)・`current`(現在盤面)・`select`(選択肢)をLLM向けテキストに変換する設計のテンプレートになる。ポケカでは「自分の場/ベンチ/手札/トラッシュ/サイド枚数」「相手の場/ベンチ/トラッシュ/サイド/手札枚数」「スタジアム・特殊状態」「直近ログ」に対応づければよい。
- **KAGはポケカで必須**: 約2000枚のカードプールに対し、LLMの内部知識だけでは効果文(ワザ・特性・トレーナーズ)を正確に再現できない。`current`に含まれる各カードの効果テキストをRAG的に整形注入し、ダメージ計算・エネルギー要件・特性発動条件を明示することで幻覚を抑える設計が極めて有効。
- **パニック行動の抑制**: 本コンペでもLLMが毎ターン一貫性のない手(無意味なエネルギー付け替え等)を返すリスクがある。自己一貫性投票はLLM呼び出しを3倍に増やすため、**10分の時間予算と勝率のトレードオフ**を測定したうえで重要局面のみ適用するのが現実的。
- **学習不要の利点**: 提出物は`main.py + deck.csv + cg/`であり、巨大モデルの同梱や学習は困難。PokeLLMon型のプロンプトのみのアプローチは、API呼び出し(許される場合)または軽量同梱モデルで実装できる現実解。
- **弱点を補う設計**: 消耗戦・読み合いの弱さは、PokeChamp型のミニマックス探索やルールベースの安全網と組み合わせることで補完すべき。LLMを「評価器」、探索を「先読み」に役割分担するのが堅実。
