# VGC-Bench: チーム条件付き MARL と被搾取性 — 競技ポケモンの統一ベンチマーク

## 概要

VGC-Bench(Angliss et al., 2026)は、競技ポケモン(VGC/Singles)を題材に、**約 \(10^{139}\) という他ゲームを凌駕するチーム構成空間**で「多様なチーム戦略を使いこなす」ことを目標に据えた統一ベンチマーク。70万件超の人間対戦ログと、ヒューリスティック・LLM・行動クローン・MARL(自己対戦/fictitious play/double oracle)など11種のベースラインを提供する。ミラーマッチではプロ競技者に勝利する一方、**チーム多様性が増すほど被搾取性が悪化**するというトレードオフを定量化した。「デッキ(チーム)とバトル方策を同時に最適化する」という PTCG AI Battle の核心課題に最も近い、ポケモン領域の最新かつ実装公開済みの研究である。

## 手法の要点

### ベンチマーク構造と progressive 評価
まず単一チームのミラーマッチで学習・評価し、次に **1 / 4 / 16 / 64 チーム**と扱うチーム集合を段階的に広げて、汎化の劣化と被搾取性を測る。クロス評価は各ペア200試合(LLMは20)、5独立試行平均でヒートマップ化。未知72チームで汎化、学習露出チームで性能を測る分離設計。

### 11ベースライン
- **MARL/PSRO 系**: pure self-play、fictitious play、double oracle、policy exploitation(経験ゲーム理論)。
- **行動クローン**(人間ログ70万件超から)、**LLM**、**ヒューリスティック**(poke-env の RandomPlayer / MaxBasePower / SimpleHeuristics)。

### 実装(poke-env + Showdown)
- **観測・行動符号化**: poke-env を介して Pokémon Showdown と連携(合法手マスクは poke-env 側で処理)。
- **チーム設定**: team string で構成し、ミラー(同一チーム同士)を除外したり、random teampreview でエピソード毎にランダム化したりできる。
- 学習エントリは `python -m vgc_bench.train` 等のモジュール、`train.sh` 等が Showdown サーバ群を自動管理。

## 主要な結果や知見

- **ミラーではプロ超え**: 単一チーム(ミラーマッチ)に限れば、最強アルゴリズムはプロ競技者級の性能を達成。
- **多様性 vs 被搾取性のトレードオフ**: 単一チームで最強の方策は、チーム集合を広げると性能が落ち**被搾取性が増す**。一方で未知チームへの汎化は向上。「特定チームへの過適合」と「汎化・頑健性」の緊張関係を明示。
- **チーム条件付けの必要性**: 最適戦略は「自分のチーム」と「相手のチーム」の両方に強く依存するため、チームに条件付けない単一方策では限界がある。double oracle / fictitious play がメタゲームの被搾取性管理に有効。
- **公開資産**: コード・データを GitHub / Hugging Face で公開し、再現可能な多エージェント戦略研究の基盤を提供。

## 本コンペ(PTCG AI Battle)への応用

VGC-Bench の「チーム(=デッキ)とバトル方策の同時最適化」は、PTCG の deck.csv 60枚+agent() を1提出で扱う構造と本質的に同じ問題であり、最も直接の手本となる。

1. **デッキ条件付き方策を学ぶ**: VGC-Bench が示す通り、最適手は自分のデッキと相手のデッキに依存する。PTCG でも方策を **deck.csv(または抽出したデッキ特徴)で条件付け**し、「このデッキならこう回す」を学習させるべき。単一デッキ過適合は被搾取性を高める。
2. **double oracle / fictitious play でメタを管理**: 自分のデッキ群と相手デッキ群の経験ゲームを構築し、double oracle で「最も搾取的な相手」を逐次追加→それに勝てる方策を追加、を繰り返すことで、提出する main 方策の被搾取性を抑えられる。これは ByteRL の教訓への直接の処方箋でもある。
3. **被搾取性 vs 汎化のトレードオフを意識した提出設計**: Kaggle は多数の未知相手と当たるため、「特定デッキで最強」より「幅広い相手に頑健」が効く可能性が高い。VGC-Bench の progressive 評価(1→4→16→64チーム)を模し、自分の方策を**多様な相手デッキ集合**で評価して汎化崖を測ってから提出する。
4. **70万ログ流の人間データ活用**: VGC-Bench は大規模人間ログで行動クローンを初期化に使う。PTCG でも人間ログがあれば BC 初期化→MARL 微調整の順で、探索空間を絞りサンプル効率を確保(Metamon/AlphaStar と同方針)。
5. **合法手マスクと10分制約**: poke-env が合法手マスクを処理するように、PTCG では渡される合法手リスト index に対し方策出力をマスクして softmax/argmax するだけでよい。探索を使わない MARL 方策なら単一 forward 推論で、10分予算内に多数の手を捌ける。
6. **実装の流用**: cameronangliss/VGC-Bench は self-play / fictitious play / double oracle / BC / LLM / heuristic の学習・評価パイプラインが揃っており、cabt Engine 用にラッパを差し替えれば、PTCG 向け MARL パイプラインの土台として再利用できる。
