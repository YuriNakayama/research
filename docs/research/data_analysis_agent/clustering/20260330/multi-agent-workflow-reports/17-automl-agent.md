## Abstract (原文)

AutoML-Agent is a multi-agent LLM framework that accelerates AI development by automating tasks in the development pipeline, such as optimal model search and hyperparameter tuning. The framework uses multiple specialized language model agents working in parallel to handle the complete machine learning workflow, from data retrieval through model deployment. Key innovations include a retrieval-augmented planning strategy and multi-stage verification for code generation. Testing across fourteen datasets demonstrated higher success rates compared to existing approaches.

## Abstract (日本語)

AutoML-Agentは、最適モデル探索やハイパーパラメータチューニング等の開発パイプラインにおけるタスクを自動化し、AI開発を加速するマルチエージェントLLMフレームワークである。複数の専門化LLMエージェントが並列に動作し、データ取得からモデルデプロイまでの完全な機械学習ワークフローを処理する。主要な革新点は検索拡張計画戦略とコード生成のためのマルチステージ検証である。14のデータセットでの評価により、既存手法と比較して高い成功率を達成した。

## 1. 概要

AutoML-Agentは、機械学習パイプラインの全工程（データ取得→前処理→モデル設計→学習→評価→デプロイ）を自動化するマルチエージェントフレームワークである。3つの技術的革新を中心に設計されている：

1. **検索拡張計画（Retrieval-Augmented Planning）**: ドメイン知識と過去のソリューションを参照した計画立案
2. **並列専門エージェント（Parallel Specialist Agents）**: タスク分解と専門エージェントによる並列実行
3. **マルチステージ検証（Multi-Stage Verification）**: 複数段階でのコード・結果検証

7つのダウンストリームタスク、14のデータセットにまたがる評価で、既存のLLMベースAutoMLフレームワークを上回る成功率を達成し、ICML 2025に採択された。

## 2. 問題と動機

### 2.1 AutoMLの現状課題

1. **パイプラインの複雑性**: 機械学習パイプラインは多段階のプロセスを含み、各段階での最適な選択の組み合わせが膨大
2. **ドメイン間の多様性**: CV、NLP、テーブルデータ、グラフ、時系列等の異なるドメインで最適なアプローチが大きく異なる
3. **専門知識の要求**: 効果的なモデル設計とハイパーパラメータチューニングには深い専門知識が必要
4. **再現性とデプロイ**: 研究レベルの成果を実運用環境にデプロイするギャップが大きい

### 2.2 既存LLMベースAutoMLの限界

- **単一計画のリスク**: 一度の計画立案に依存し、最適でない計画のリカバリーが困難
- **逐次実行の非効率**: エージェントが逐次的に動作し、独立タスクの並列化がされていない
- **検証の不足**: 生成されたコードやモデルの検証が不十分で、エラーが伝播する
- **限定的なタスクカバレッジ**: 特定のタスクタイプのみを対象とし、汎用性に欠ける

### 2.3 研究目標

- データ取得からモデルデプロイまでの完全自動化
- 多様なMLタスク・ドメインへの汎化
- 効率的な並列実行と品質保証の両立

## 3. 提案手法

### 3.1 検索拡張計画（Retrieval-Augmented Planning）

従来の単一計画立案ではなく、検索拡張により探索を強化する：

- **ドメイン知識の検索**: タスクに関連する過去のソリューション、ベストプラクティス、技術文書を参照
- **複数計画の探索**: 単一の最適計画ではなく、複数の有望な計画を生成・評価
- **適応的計画修正**: 実行結果に基づく計画の動的修正
- **情報に基づく意思決定**: LLMの出力を関連技術情報でグラウンディング

### 3.2 並列専門エージェント

複雑なタスクを独立したサブタスクに分解し、専門エージェントが並列に実行：

**エージェント構成**:

| エージェント | 役割 |
|------------|------|
| Prompt Agent | 指示チューニングLoRAアダプター（Mixtral-8x7B）によるプロンプト最適化 |
| Data Agent | データセット処理（取得、前処理、特徴量エンジニアリング） |
| Model Agent | モデル選択・設計（アーキテクチャ探索、ハイパーパラメータ設定） |
| Operation Agent | パイプライン実行の調整 |
| Agent Manager | マルチエージェントインタラクションのオーケストレーション |

**並列実行の利点**:
- 独立タスクの同時処理による時間効率の向上
- 各エージェントの専門化によるソリューション品質の改善
- 失敗した個別タスクの独立した再試行が可能

### 3.3 マルチステージ検証

パイプラインの複数段階で検証を実施：

1. **計画段階**: 計画の実行可能性と整合性の検証
2. **コード生成段階**: 生成コードの構文チェックと実行テスト
3. **実行段階**: 中間結果の妥当性検証
4. **出力段階**: 最終モデルの性能検証とデプロイ準備状態の確認

各段階でエラーが検出された場合、対応するエージェントにフィードバックを送り、修正を促す。

## 4. アーキテクチャ

### 4.1 4段階パイプライン

```
Stage 1: 初期化（Initialization）
    - ユーザー指示の検証
    - タスク種別の特定
    - 制約条件の確認
    ↓
Stage 2: 計画（Planning）
    - ML情報の抽出
    - 検索拡張による計画立案
    - 複数実行戦略の生成
    ↓
Stage 3: 実行（Execution）
    - サブタスク分解
    - 並列専門エージェントによる実行
    - マルチステージ検証
    ↓
Stage 4: 出力（Output）
    - デプロイ準備完了コードの生成
    - 最終検証と品質保証
```

### 4.2 エージェント間協調構造

```
Agent Manager（オーケストレーター）
    ├── Prompt Agent（Mixtral-8x7B LoRA）
    ├── Data Agent
    │     ├── データ取得
    │     ├── 前処理
    │     └── 特徴量エンジニアリング
    ├── Model Agent
    │     ├── アーキテクチャ探索
    │     ├── ハイパーパラメータ設定
    │     └── モデル構築
    └── Operation Agent
          ├── パイプライン実行
          ├── 検証
          └── デプロイ
```

### 4.3 技術スタック

- **LLMバックエンド**: GPT-4, GPT-3.5-turbo, Mixtral-8x7B（LoRA）
- **推論エンジン**: vLLM 0.4.1
- **言語**: Python 3.11
- **環境**: conda

| タスクカテゴリ | ドメイン | データセット数 |
|-------------|---------|-------------|

| 時系列予測 | Time Series | 2 |

### 6.2 評価指標

- **分類タスク**: Accuracy, F1-score
- **回帰タスク**: RMSLE, RMSE
- **クラスタリングタスク**: Rand Index (RI)
- **全体**: パイプライン完了成功率

### 6.3 主要実験結果

- **タスクカバレッジ**: 7つの異なるMLタスクカテゴリにわたる14データセットで評価
- **成功率**: 既存のLLMベースAutoMLフレームワーク（単一エージェント型含む）と比較して全般的に高い成功率
- **汎化能力**: CV、NLP、テーブル、グラフ、時系列という多様なドメインにわたる一貫した性能
- **デプロイ準備**: 生成されたコードが直接デプロイ可能な品質

### 6.4 アブレーション分析

- **検索拡張計画の効果**: RAGなしの計画立案と比較して、より最適な計画の発見率が向上
- **並列実行の効果**: 逐次実行と比較して処理時間の大幅な短縮
- **マルチステージ検証の効果**: 単一段階検証と比較してエラー伝播の抑制と最終成功率の向上
- **エージェント専門化の効果**: 汎用エージェントと比較して各タスクでの品質向上

### 6.5 LLMバックエンドの影響

- GPT-4バックエンドが全般的に最高性能
- GPT-3.5-turboでも競争力のある成功率を達成
- Mixtral-8x7B（LoRA）によるPrompt Agentがコスト効率を改善

## 7. 備考

### 7.1 学術的意義

AutoML-AgentはICML 2025に採択され、LLMベースのAutoMLフレームワークとして包括的な評価を実施した先駆的研究である。データ取得からデプロイまでの「フルパイプライン」AutoMLという概念を確立し、検索拡張計画・並列専門エージェント・マルチステージ検証の3つの技術的貢献を明確に示した。

### 7.2 データ分析エージェントへの示唆

- **並列専門エージェント**: データ分析パイプラインの各段階（前処理、EDA、モデリング、評価、可視化）を専門エージェントに分担させる設計パターンとして直接適用可能
- **検索拡張計画**: 過去の分析事例やベストプラクティスを参照した分析計画の立案に応用可能
- **マルチステージ検証**: 分析結果の妥当性を複数段階で検証し、誤った分析結果の伝播を防止
- **フルパイプライン自動化**: データ取得から最終レポートまでの一貫した自動化のモデルケース

### 7.3 制限事項

- GPT-4への依存度が高く、APIコストが高額になる可能性
- 14データセットは広範だが、各カテゴリ内のデータセット数が少ない（特にNLP）
- デプロイ段階の検証が限定的（実運用環境でのテストが不足）
- vLLM 0.4.1への厳密な依存が環境構築の障壁となる可能性

### 7.4 産業的インパクト

ICML 2025採択は、マルチエージェントAutoMLの産業応用への道を開くものである。特に、プログラミング知識の限られたドメイン専門家がMLパイプラインを構築できるようにする点で、機械学習の民主化に貢献する。検索拡張計画は、企業内の過去のMLプロジェクトの知見を活用した効率的な開発を可能にする。
