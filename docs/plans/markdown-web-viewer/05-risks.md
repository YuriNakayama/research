# Markdownウェブビューア — リスクと依存関係

## リスク一覧

| # | リスク | 影響 | 確率 | 軽減策 |
|---|--------|------|------|--------|
| 1 | Markdownレンダリング品質 — 数式・Mermaid図・複雑なテーブルが期待通りにレンダリングされない | 高 | 高 | 段階的実装: 基本Markdown → GFMテーブル → KaTeX数式 → コードハイライト。各段階で実際のレポートファイルでテスト。Mermaidは初回スコープ外にする選択肢も |
| 2 | Amplify + Terraform統合 — Amplify Gen2の一部機能がTerraform未サポートで手動設定が必要になる | 高 | 中 | Terraform importの予備計画を用意。管理できない部分はコンソールで設定し、手順を文書化。将来的にTerraformに取り込む |
| 3 | DynamoDB同期の信頼性 — ビルド時同期が失敗した場合、一覧ページのデータが古いままになる | 中 | 低 | ファイルシステムフォールバック機構を実装。DynamoDB接続失敗時はCSV/MDファイルから直接データを取得する |
| 4 | Amplifyビルド時間 — レポート数増加に伴いSSGビルド時間が長くなる | 低 | 中 | ISRで段階的再生成。全ページSSGではなく、一覧ページはSSR、詳細のみSSG+ISRにする |
| 5 | rehype-sanitize と KaTeX の競合 — XSS対策のsanitizeがKaTeXのHTML属性を除去してしまう | 中 | 高 | rehype-sanitizeのスキーマをカスタマイズし、KaTeXが使用するclass属性・style属性を明示的に許可する |
| 6 | Cognito UIのカスタマイズ制限 — @aws-amplify/ui-reactのログインUIがデザインに合わない | 低 | 中 | Amplify UIのテーマカスタマイズ機能を活用。最悪の場合はHeadless UIで独自ログインフォームを構築 |

## 外部依存関係

| 依存先 | 内容 | リスク |
|--------|------|--------|
| AWS Amplify | ホスティング、ビルド、CDN配信 | サービス障害時にサイト全体が停止。SLA 99.9% |
| AWS Cognito | ユーザー認証 | 障害時にログイン不可。招待制のため影響範囲は限定的 |
| AWS DynamoDB | レポートメタデータ | 障害時にファイルシステムフォールバックで一覧表示は維持 |
| GitHub | リポジトリ、CI/CD | Amplifyビルドトリガーが動作しない。手動ビルドで対応可能 |
| KaTeX | 数式レンダリング | クライアントサイドライブラリのため、CDN障害の影響なし |
| Shiki | コードハイライト | ビルド時生成のため、ランタイム依存なし |

## 技術的負債

- **初期段階でMermaid図非対応**: レンダリング複雑性のため、初回リリースではMermaid図をコードブロックとして表示。将来的にmermaid.jsを統合
- **DynamoDB同期がビルド時のみ**: リアルタイム同期ではないため、パイプライン実行後Amplifyビルド完了まで数分のラグがある
- **Terraform + コンソール混在の可能性**: Amplifyの一部設定がTerraformで管理できない場合、コンソール設定が残る

## オープンアイテム

- カスタムドメインの設定（Route 53 or 外部DNS）
- Amplifyのビルドインスタンスサイズ（Standard / Large）の決定
- Cognito招待ユーザーの初期パスワード配布方法
- レポート数が1000件を超えた場合のビルド時間とパフォーマンスの検証
