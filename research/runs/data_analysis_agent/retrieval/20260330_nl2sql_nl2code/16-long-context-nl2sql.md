# Is Long Context All You Need? Leveraging LLM's Extended Context for NL2SQL

- **Link**: https://arxiv.org/abs/2501.12372
- **Authors**: Yeounoh Chung, Gaurav T. Kakkar, Yu Gan, Brenton Milne, Fatma Ozcan
- **Year**: 2025
- **Venue**: VLDB 2025
- **Type**: Academic Paper

## Abstract

We investigate how extended context windows in large language models can enhance natural language to SQL (NL2SQL) translation. We examine performance and latency tradeoffs using Google's Gemini-1.5-Pro, testing various contextual inputs including column example values, question-SQL pairs, user hints, and database documentation. Our key findings show that long context LLMs are robust and do not get lost in the extended contextual information. We demonstrate that our pipeline achieves strong benchmark results without fine-tuning or computationally expensive consistency techniques. This represents the first systematic investigation of how extended context windows and additional contextual data impact both accuracy and latency in NL2SQL generation tasks.

## Abstract（日本語訳）

我々は、大規模言語モデルの拡張コンテキストウィンドウが自然言語からSQL（NL2SQL）への変換をどのように強化できるかを調査する。Google Gemini-1.5-Proを使用して、カラムサンプル値、質問-SQLペア、ユーザーヒント、データベースドキュメントなどの様々なコンテキスト入力について性能とレイテンシのトレードオフを検証する。主要な発見として、ロングコンテキストLLMは堅牢であり、拡張されたコンテキスト情報の中で迷子にならないことが示された。我々のパイプラインはファインチューニングや計算コストの高い一貫性技法なしに強力なベンチマーク結果を達成することを実証する。これは、拡張コンテキストウィンドウと追加のコンテキストデータがNL2SQL生成タスクの精度とレイテンシの両方にどのように影響するかについての初の体系的調査である。

## 概要

本研究は、大規模言語モデルの拡張コンテキストウィンドウ（最大200万トークン対応のGemini-1.5-Pro）を活用してNL2SQL変換の精度を向上させる手法を体系的に調査した初の研究である。従来のNL2SQLアプローチでは、限られたコンテキストウィンドウに収めるためにスキーマリンキングやテーブルフィルタリングなどの前処理が不可欠であったが、これらの前処理自体がエラーの原因となりうるという問題があった。本研究は「情報をフィルタリングして限られたウィンドウに収める」のではなく「豊富な情報をすべてコンテキストに含める」というアプローチを提案する。3段階のエージェントワークフロー（Generate → Fix & Rewrite → Verify）を採用し、完全なデータベーススキーマ、カラムサンプル値、ユーザーヒント、合成例、SQLiteリファレンスドキュメントなど6種類のコンテキスト入力の効果を体系的に評価した。BIRDベンチマークで67.41%、Spider 1.0で87.10%の精度を達成し、ファインチューニングやself-consistencyなしに競争力のある結果を示した。また、コンテキストサイズとレイテンシの間に近線形の相関（R²=92.6%）があることを発見し、32Kトークンを超えるとレイテンシの分散が大幅に拡大するという実用的な知見も提供している。

## 問題設定

- **スキーマリンキングのエラー伝播**: 従来のNL2SQLパイプラインでは、限られたコンテキストウィンドウに収めるためにテーブル・カラムの事前選択（スキーマリンキング）が必要であるが、この前処理段階でのエラーが後続のSQL生成に伝播し、回復不能な精度低下を引き起こす。

- **コンテキスト情報の活用不足**: データベースのカラムサンプル値、業務ドキュメント、類似クエリ例など、SQL生成に有用な補助情報が存在するにもかかわらず、コンテキストウィンドウの制約により十分に活用できていない。

- **ロングコンテキストの効果の未検証**: LLMのコンテキストウィンドウが急速に拡大している（100万〜200万トークン）にもかかわらず、NL2SQLタスクにおけるロングコンテキストの効果と限界について体系的な調査が行われていない。

- **精度とレイテンシのトレードオフ**: より多くのコンテキストを提供することで精度は向上する可能性があるが、推論レイテンシとコストも増大する。このトレードオフの定量的な理解が不足している。

- **ファインチューニングなしの性能限界**: ファインチューニングやself-consistencyなどの計算コストの高い手法を使用せずに、どの程度の性能を達成できるかが不明確であった。

## 提案手法

### 3段階パイプラインアーキテクチャ

**Stage 1: Generate（生成）**
- 完全なデータベーススキーマ、カラムサンプル値、ユーザーヒント、合成例をコンテキストに含めてSQL生成
- スキーマリンキングを省略し、すべてのテーブル情報を提供（再現率100%を保証）

**Stage 2: Fix & Rewrite（修正・書き換え）**
- 生成されたSQLの自己修正（Self-correction）
- 曖昧性解消（Disambiguation）による改善
- スキーマ情報を用いた出力正確性検証

**Stage 3: Verify（検証）**
- 生成SQLの実行可能性と結果の妥当性を検証
- エラー検出時のリトライメカニズム

### コンテキスト入力の6カテゴリ

1. **データベーススキーマ**: ターゲットデータベースの完全なテーブル定義
2. **カラムメタデータ**: 非自明なテキストカラムからのサンプル値
3. **ユーザー明確化情報（ヒント）**: カラム参照や数学的表現に関する指示
4. **In-context例**: 類似性ベースの検索例および合成生成された質問-SQLペア
5. **ドキュメント**: SQLiteリファレンスマテリアル（粗粒度・細粒度チャンク）
6. **検証データ**: 出力正確性検証用のスキーマ情報

### 評価指標

- **実行精度（Execution Accuracy）**: 生成SQLの実行結果が正解と一致する割合
- **レイテンシ**: 推論にかかる時間（正規化レイテンシで比較）
- **トークン数**: 各設定での平均累積トークン数

## 実験・評価

### セットアップ

- 主要モデル: Google Gemini-1.5-Pro（最大200万トークンコンテキスト）
- 比較モデル: Gemini-1.5-Flash（レイテンシ/コスト比較用）
- 評価ベンチマーク: BIRD dev、Spider 1.0、KaggleDBQA、BEAVER dw
- 評価指標: 実行精度（EX）、正規化レイテンシ、トークン数

### 主要結果

**ロングコンテキストの頑健性**:
LLMは拡張されたコンテキスト情報の中で「迷子にならない」ことが実証された。全データベーステーブルを含む設定（72,619トークン）でも精度低下はほぼなく、62.58%を維持。

**コンテキスト入力の影響度**:
- ユーザーヒントが最大の改善効果（+16.36%）を提供
- 自己修正（+2.81%）とカラム値（+1.76%）が次に有効
- 合成例（+1.05%）と曖昧性解消（+0.71%）は限定的な改善

**スキーマリンキングの不要性**:
全テーブル包含（再現率100%）はTop-7検索（再現率82%）を7.63ポイント上回り、スキーマリンキングの省略が精度向上につながることを実証。

**Many-Shot ICLの限界**:
類似検索による訓練データからの例示は「無視できる程度の改善」かつ時に性能低下。合成例はBIRD dev（クロスドメイン設定）で検索例を大幅に上回るが、KaggleDBQA（同一ドメイン）では劣る。

**レイテンシとコスト分析**:
- コンテキストサイズとレイテンシに近線形相関（R²=92.6%）
- 32Kトークンを超えると分散が大幅に拡大
- フルパイプラインの正規化レイテンシはベースラインの37.1倍
- Gemini-1.5-Flashはレイテンシ20%削減、コスト94%削減を達成（精度は66.49% vs 67.41%で微減）

**リーダーボードでの位置づけ**:
ファインチューニングなし・self-consistencyなしで67.41%を達成し、ファインチューニング+self-consistency利用のDistillery（67.21%）を上回る。ただし、同手法を利用するCHASE-SQL（74.46%）やXiYan-SQL（73.34%）とは約7ポイントの差がある。

## 備考

- 本研究は「ロングコンテキストがあればスキーマリンキングは不要か?」という根本的な問いに対して肯定的な実証を提供しており、NL2SQLパイプラインの設計思想に大きな影響を与える
- 「情報をフィルタリングする」から「すべての情報を提供する」へのパラダイム転換は、LLMのコンテキストウィンドウ拡大に伴い今後さらに重要性を増す
- レイテンシとコストのトレードオフ分析は実運用環境での導入判断に直接的に有用な知見を提供している
- 32Kトークンの臨界点の発見は、コンテキスト最適化の実践的指針となる
- VLDB 2025に採択されており、データベースコミュニティにおける高い評価を受けている
- Gemini-1.5-Pro/Flashに限定した評価であり、他のロングコンテキストLLM（Claude、GPT-4等）での一般化可能性は今後の課題である
